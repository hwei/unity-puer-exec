# 0010 T1.2.1 Formal Package Home

## Task

- `T1.2.1 Create Formal Package Home In unity-puer-exec`

## Goal

Move the Unity-side package source out of the validation host and create its first formal home inside `unity-puer-exec/`, while keeping the current Unity-side capability baseline intact.

This plan only covers the productized repository. Rewiring the validation host to consume the migrated package is tracked separately under `T1.2.2`.

## Scope

In scope:

- create a formal package directory in `unity-puer-exec/`
- migrate the current Unity package source from the validation host into that directory
- rename package metadata from `com.c3.unity-puer-exec.validation` to `com.txcombo.unity-puer-exec`
- rename Unity-side assembly and namespace identity away from `C3.UnityPuerExecValidation`
- preserve the current editor-host capability baseline during the move
- document any intentionally transitional validation-oriented APIs that remain after migration

Out of scope:

- changing the validation host to consume the migrated package
- redesigning the CLI surface
- deciding CLI packaging or AOT strategy
- fully restructuring the Unity package into its final production layout beyond what is required for the migration

## Proposed Package Home

- package root: `packages/com.txcombo.unity-puer-exec/`
- initial Unity layout:
  - `package.json`
  - `Editor/`
  - editor asmdef
  - migrated runtime bridge/server code

This keeps the first migration small and leaves deeper package layout formalization to `T1.3`.

## Work Steps

1. Inspect the validation-host package and record the exact files, asmdef identity, namespace identity, and validation-only wording that must change during migration.
2. Create `packages/com.txcombo.unity-puer-exec/` inside `unity-puer-exec/` with the minimal Unity package structure required to host the migrated code.
3. Copy the current package source from the validation host into the new package home without changing behavior beyond identity migration.
4. Update `package.json` to formal product identity, including name, display name, description, and author wording as needed for the productized repository.
5. Rename the editor asmdef and root namespace from the validation-only identity to the formal `UnityPuerExec` identity, keeping compilation references intact.
6. Review the migrated C# surface and separate naming cleanup into two classes:
   - required now for formal package identity
   - intentionally transitional and deferred to `T1.3`
7. Update repository-facing docs only where needed so the new package location is discoverable and clearly identified as the source of truth.
8. Validate that the productized repository now contains the formal package home with no remaining dependency on the validation host as source storage for this code.

## Validation

- `T1.2.1` does not require end-to-end execution inside the validation host; host-wired runtime validation is deferred to `T1.2.2`
- verify the migrated package tree exists under `unity-puer-exec/packages/com.txcombo.unity-puer-exec/`
- inspect the migrated `package.json`, asmdef, and namespaces to confirm the formal identity is in place
- run any repository-level checks that can validate the migration without requiring `T1.2.2` to be complete
- perform a targeted diff review against the validation-host source to confirm capability-preserving migration rather than accidental behavioral rewrite

## Risks

- the current `UnityPuerExecServer.cs` mixes formal host capability with validation-oriented entry points, so the migration can easily overreach into `T1.3` if not kept disciplined
- naming cleanup may affect Unity bridge type names that are implicitly referenced by JavaScript snippets or external tooling
- if package-root assumptions are baked into repository scripts or tests, migration may require small follow-up doc or test adjustments

## Exit Criteria

- `unity-puer-exec/` contains the first formal home of `com.txcombo.unity-puer-exec`
- migrated Unity-side source no longer uses the validation-only package identity
- any remaining transitional validation-oriented APIs are explicitly noted rather than left ambiguous
- `T1.2.2` is unblocked to switch the validation host over to consuming the migrated package
