# Plan: T1.2.2.1 Rewire Validation Host Manifest To Local Package

## Scope

Define and apply the validation host workflow in which the host first returns to a clean baseline from the `unity-puer-exec` fork point, then locally injects the formal package from `unity-puer-exec/packages/com.txcombo.unity-puer-exec/` by editing `Project/Packages/manifest.json`.

This plan covers host wiring and operating-model documentation only. It does not require end-to-end Unity execution; minimal host runtime validation is deferred to `T1.2.2.2`.

## Intended Outcome

- The validation host operating model identifies a clean baseline branch or commit from the `unity-puer-exec` fork point.
- The validation host manifest can be locally edited to point at the formal local package.
- Documentation explains that this host wiring is a local-only injection workflow and should not become a normal host commit stream.

## Steps

1. Inspect the validation host package wiring surface.
   - Identify the clean host baseline branch or commit that represents the `unity-puer-exec` fork point without the formal package injected.
   - Read `c3-client-tree2/Project/Packages/manifest.json`.
   - Read `c3-client-tree2/Project/Packages/packages-lock.json` only to understand the current host package resolution shape.
   - Confirm how the old `com.c3.unity-puer-exec.validation` package currently appears in the host so the new local-only injection path replaces it cleanly.

2. Rewire the validation host to the formal local package.
   - Starting from the clean host baseline, edit `c3-client-tree2/Project/Packages/manifest.json` locally so it points to `unity-puer-exec/packages/com.txcombo.unity-puer-exec/` through a local file dependency.
   - Prefer a reproducible relative path from the host project over a machine-specific absolute path.
   - Do not require a committed `packages-lock.json` change for this task boundary; treat lockfile updates as Unity-derived follow-up material for `T1.2.2.2` unless a narrower issue is discovered.

3. Distill the resulting host operating model.
   - Update long-lived docs in `unity-puer-exec/` so readers can find the clean-baseline expectation and the local-only manifest injection workflow.
   - Clarify that the host manifest edit is for local validation use and is not the product repository's normal commit path.

4. Run direct validation for this task boundary.
   - Verify the identified host baseline is documented clearly enough to reproduce.
   - Verify the local manifest edit points to the formal package with a reproducible local path.
   - Verify the plan and long-lived docs describe the manifest injection as local-only host wiring.
   - Treat `packages-lock.json` as observational only for this task unless a concrete manifest-wiring inconsistency forces a narrower follow-up.
   - Do not treat Unity import or runtime execution as part of this task; that belongs to `T1.2.2.2`.

## Risks To Watch

- The exact clean baseline may be ambiguous if the fork-point branch or commit is not already documented clearly.
- The validation host may contain other files or scripts that still reference the old package name.
- Unity may rewrite the lockfile shape after import, so any lockfile handling should stay deferred until `T1.2.2.2`.
- Relative path calculation can drift if the host/project working directory assumption is wrong.

## Exit Criteria

- The host baseline expectation from the `unity-puer-exec` fork point is documented.
- `c3-client-tree2/Project/Packages/manifest.json` can be locally edited to consume the formal package through a reproducible local path.
- Long-lived docs point readers to the clean-baseline plus local manifest injection workflow.
- Static validation for the rewiring passes, with runtime validation explicitly deferred to `T1.2.2.2`.
