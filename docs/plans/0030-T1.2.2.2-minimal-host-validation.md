# Plan: T1.2.2.2 Run Minimal Host Validation Against Local Package

## Scope

This task will prove that the validation host can consume the formal local package through manifest injection and run one minimal Unity-side validation path.

This task will not formalize the broader end-to-end validation entry, replace the current runtime path with the future CLI contract, or decide how host-local follow-up artifacts such as `packages-lock.json` should be committed.

This task may use a temporary agent-operated validation workflow with a background Unity process and manual log observation. It does not need to adopt the repository's more formal automated session flow as part of this proof step.

## Requirements

- Requirement: `T1.2.2.2` must prove host consumption of the formal package by running against a rewired host manifest, not by reasoning from static files alone.
- Scenario: starting from the host workspace layout described in `0005-validation-host-local-package-injection.md`, the implementer first moves the validation host checkout to commit `e891b5495e8a8ae1950a41d0d85f6314676e13e9` or records a concrete reason why a specific clean descendant is equivalent, then rewires `manifest.json` to `com.txcombo.unity-puer-exec`, launches Unity against the host project, and invokes a minimal batch entry from the formal package.
- Requirement: the chosen runtime path should be the narrowest path that proves package import and execution.
- Scenario: the validation should prefer the existing `UnityPuerExecBatch.PrintHealth` entry unless the rewired host shows that this path is no longer viable.
- Requirement: the validation plan must assume a cold host startup is possible when the validation host has no prebuilt `Library/` state.
- Scenario: the implementer allows substantially longer startup time than an already-warmed project, treats initial package import and compilation as expected work instead of immediate failure, and distinguishes timeout policy from runtime failure evidence.
- Requirement: the success criteria must name the exact runtime markers that prove the formal package code executed inside the host.
- Scenario: a successful `PrintHealth` run emits all three expected markers from the formal package batch entry: `[UnityPuerExecBatch] health-check-start`, `[UnityPuerExecBatch] port=...`, and `[UnityPuerExecBatch] health-check-end`.
- Requirement: the validation posture must tolerate long-running cold startup better than a single short blocking command.
- Scenario: the implementer starts Unity in the background with an explicit `-logFile` path, periodically checks whether that log keeps growing and what the latest lines say, and only kills and retries after observing a clear error or a convincing lack of progress.
- Requirement: the validation evidence channel must be explicit and stable for a background run.
- Scenario: this task treats the explicit Unity `-logFile` target as the source of truth for startup progress and health markers, rather than relying on attached stdout.
- Requirement: if `PrintHealth` cannot be used, any substitute entry point must still be a formal-package method invoked through the rewired host and must emit equally specific success evidence before the task can claim success.
- Requirement: durable documentation should record the exact minimal validation command and any host-local caveat still in force after the run.

## Risks

- The host may still carry the legacy validation package directory, and Unity resolution behavior may expose ambiguity even after manifest rewiring.
- The recorded fork-point baseline may require local recovery work in the validation host before the minimal proof path can run, so any deviation to a clean descendant must be justified explicitly rather than assumed.
- The formal package may import cleanly but fail at runtime because a dependency assumption was previously satisfied only by the host-carried package layout.
- A cold host with no `Library/` can spend significant time on import and compilation before reaching the batch entry, so naive timeout settings can produce misleading failure signals.
- Unity may rewrite host-local derived files during validation; only stable conclusions, not incidental local artifacts, should be committed from this task.

- A minimal formal-package runtime path has succeeded, or the task has been explicitly converted into a documented blocker with concrete failure evidence.
