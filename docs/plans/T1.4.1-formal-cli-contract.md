# Plan: T1.4.1 Define Formal CLI Contract And Command Tree

## Scope

Define the first formal CLI contract for `unity-puer-exec` without changing implementation language, packaging strategy, or code location yet.

This task should decide the formal command tree, command naming, parameter-shape principles, output and exit-code contract, and the workflow-oriented help requirements that later implementation work must satisfy.

This task should not implement the new contract in code, migrate the current CLI implementation out of the skill-owned location, or decide whether the final CLI remains Python-based or is replaced by a self-contained executable.

This task may identify redundant commands, missing formal capabilities, or contract gaps in the current baseline and may resolve them at the contract level, but any implementation changes remain deferred to later tasks.

## Requirements

1. The formal contract must define one primary CLI entry name: `unity-puer-exec`.

2. The formal contract must define a single flat command tree under that entry rather than introducing an additional command group such as `session`.

3. The formal contract must cover current user-visible capability areas already present in the baseline:
   - session readiness and recovery waiting
   - log-pattern waiting
   - remote JavaScript execution
   - async result retrieval

4. The formal contract must explicitly decide whether current command names and argument names remain formal, are renamed, or require aliases during later implementation work.

5. The formal contract must define the normative workflow boundary between readiness-oriented commands and execution-oriented commands, including whether commands such as `exec` assume a ready Unity session, establish readiness implicitly, or require an explicit prior readiness step.

6. The formal contract must define, per command, which runtime precondition inputs are part of the formal surface, including items such as project-path resolution, Unity executable override, keep-alive behavior, and timeout controls.

7. The formal contract must explicitly decide the JavaScript input contract for execution commands, including whether script content is accepted from stdin, file input, inline arguments, or a combination of those forms, with preference given to input forms that avoid shell-escaping fragility for AI-generated scripts.

8. The formal contract must define a stable output model that an AI agent can parse without relying on repository skill docs, including command-by-command stdout and stderr usage and the payload shape expected for success, expected non-success machine states, and unexpected failures.

9. The formal contract must define a baseline exit-code model, including which non-zero exits are part of the formal surface and what they mean.

10. The formal contract must define what top-level and per-command `--help` must communicate, including at least one typical usage workflow that explains how the commands fit together in practice.

11. The formal contract must explicitly decide the transitional status of `unity-puer-session`, including whether it remains as an alias during migration, what compatibility it must keep, and what later tasks are allowed to remove.

12. The formal contract must stay implementation-language-agnostic so that `T1.5` can still decide whether the final CLI is an evolved migration of the current baseline or a replacement implementation.

## Work Items

1. Inventory the current CLI surface.
   - Goal: Extract the currently observable commands, argument shapes, output payloads, and exit codes from the skill-owned baseline and tests.
   - Write scope: plan notes and later durable contract doc only.
   - Parallelism: no; this is the foundation for the later contract decisions.

2. Define the formal top-level command tree and naming model.
   - Goal: Decide the flat `unity-puer-exec` command list, including whether current names such as `wait-until-recovered` and `get-result` remain formal or should change.
   - Write scope: durable contract doc and roadmap/status output pointers only.
   - Parallelism: limited; naming choices depend on the inventory.

3. Define the formal parameter-shape rules.
   - Goal: Decide formal per-command inputs and parameter-shape rules, including argument naming, units, project-path resolution, Unity executable selection, keep-alive behavior, and timeout controls.
   - Write scope: durable contract doc only.
   - Parallelism: limited; depends on command-tree decisions.

4. Define the normative execution workflow boundary.
   - Goal: Decide how readiness-oriented commands and execution-oriented commands relate in practice, including whether `exec` requires an already-ready session, may establish readiness implicitly, or should remain invalid until explicit readiness has been established.
   - Write scope: durable contract doc only.
   - Parallelism: limited; depends on command-tree decisions.

5. Define the formal JavaScript input contract.
   - Goal: Decide whether execution commands should formally prefer stdin input, file input, inline arguments, or some combination, and decide whether any inline `--code` form remains only as transitional compatibility.
   - Write scope: durable contract doc only.
   - Parallelism: limited; depends on the execution-workflow decision.

6. Define the formal output and exit-code contract.
   - Goal: Decide command-by-command payload fields, stdout/stderr conventions, machine-state handling, and exit codes that are stable enough to constrain later implementation work.
   - Write scope: durable contract doc only.
   - Parallelism: limited; depends on the inventory and command-role decisions.

7. Define the workflow-oriented help contract.
   - Goal: Specify what `unity-puer-exec --help` and command-specific `--help` must tell an agent, including typical workflows such as ensuring Unity readiness before `exec`, using `get-result` after long-running jobs, and using recovery or log waiting when direct execution is not yet appropriate.
   - Write scope: durable contract doc only.
   - Parallelism: yes, after the command tree is stable.

8. Define the transitional entrypoint decision.
   - Goal: Decide how `unity-puer-session` behaves during migration, including whether it remains as an alias and what compatibility guarantees later implementation tasks must preserve or may remove.
   - Write scope: durable contract doc only.
   - Parallelism: limited; depends on the formal command tree and workflow decisions.

9. Distill the formal CLI contract into long-lived repo documentation.
   - Goal: Produce `docs/decisions/0007-formal-cli-contract.md` so future implementation tasks can treat it as the authoritative CLI contract baseline.
   - Write scope: `docs/decisions/0007-formal-cli-contract.md` plus roadmap/status output pointers.
   - Parallelism: no; this is the integration step for the task.

## Integration

Main agent owns the contract synthesis and the durable decision output. No code merge is expected in this task; integration means reconciling the inventory, naming decisions, output model, and help requirements into one coherent contract document.

## Validation

- Validation owner: main agent, with one clean-context subagent review before human review.
- Direct checks:
  - every currently exposed capability area is accounted for in the proposed formal command tree
  - the resulting contract makes an explicit normative decision about the relationship between readiness commands and `exec`
  - the resulting contract defines formal per-command precondition inputs rather than only general naming principles
  - the resulting contract explicitly decides the preferred JavaScript input forms for execution commands
  - the resulting contract defines command-by-command stdout/stderr and payload expectations
  - the resulting contract explicitly decides the transitional fate of `unity-puer-session`
  - the plan clearly separates contract definition from later implementation and packaging work
  - the resulting durable output can directly constrain `T1.4.2`, `T1.4.3`, and `T1.5`
- Deferred checks:
  - actual CLI behavior changes, help text changes, and import-path migration are deferred to later tasks

## Risks

- The current baseline may contain incidental names or parameter forms that are easy to preserve by habit even if they are not the clearest formal contract.
- If the output contract is underspecified now, later migration or rewrite work may accidentally diverge across implementations.
- If the workflow guidance is too abstract, `--help` may remain technically complete but still not be self-discoverable enough for an AI agent.
