# Plan: T1.4.1.1 Revise CLI Session Discovery And Lifecycle Contract

## Plan Type

Governance Plan

## Scope

Revise `docs/decisions/0007-formal-cli-contract.md` so the CLI session discovery model, command lifecycle boundaries, and shutdown semantics are stable before `T1.4.2` starts baseline implementation work.

This task should distill the already discussed command-model changes into the durable contract. It should update the authoritative CLI contract, plus any roadmap or status pointers needed to keep later work aligned.

This task should not implement the revised contract in code, move the CLI out of the skill-owned location, or decide the final packaging strategy.

## Agreed Constraints

- `docs/decisions/0007-formal-cli-contract.md` is the authoritative contract to revise in this task.
- `docs/decisions/` is governance-facing output, so this task uses a `Governance Plan`.
- CLI session selection has two mutually exclusive modes:
  - `--project-path`
  - `--base-url`
- In `--project-path` mode, project-local discovery under the Unity project `Temp/` directory is the normal session discovery path.
- The project-local discovery artifact path is `<project>/Temp/UnityPuerExec/session.json`.
- The session discovery artifact does not need to repeat `project_path`.
- The session discovery artifact should at least carry:
  - `base_url`
  - `unity_pid`
  - `written_at`
  - `protocol_version`
- Additional session artifact fields are out of scope for this task and are non-contractual best-effort data unless promoted by a later decision.
- `written_at` exists for stale-session checks and diagnostics.
- `--base-url` is a first-class direct-service addressing mode, not a debug-only path.
- In `--base-url` mode, the CLI does not own Unity Editor startup.
- `exec` may implicitly prepare Unity only in `--project-path` mode.
- `wait-until-ready` should replace the current split between `ensure-ready` and `wait-until-recovered`.
- `wait-until-ready` is a formal top-level command in the CLI surface.
- `wait-until-ready` should be documented as a specialized shortcut over the `exec` readiness path rather than as a second independent capability model.
- `wait-until-ready` returns `recovered` when the target is already healthy or becomes healthy.
- `get-result` must not restart or re-provision Unity.
- `wait-for-log-pattern` is observation-oriented, not a launch command.
- The revised contract should add `get-log-source` rather than `get-log-path` so direct-service and future non-file-backed observation sources fit the same surface.
- `get-log-source` supports the same mutually exclusive selectors and returns the current observable log source description for that selector target.
- In `--base-url` mode, `wait-for-log-pattern` and `get-log-source` may describe only the current best-effort observable source and do not imply stable server identity across calls.
- A new command is needed to guarantee stopped-state semantics, named `ensure-stopped`.
- `ensure-stopped` supports both selectors, but they remain mutually exclusive.
- In `--project-path` mode, `ensure-stopped` may:
  - inspect whether the target is stopped
  - wait until a timeout and then kill
  - kill immediately
- In `--base-url` mode, `ensure-stopped` may inspect state only and must not kill.
- Graceful shutdown is out of scope for `ensure-stopped`; callers can use `exec` to send a graceful shutdown script when that behavior is wanted.
- The revised contract must keep machine-readable status semantics for:
  - `session_missing`
  - `session_stale`
  - `address_conflict`
  - `no_observation_target`
- `--keep-unity` should leave the formal contract, though later implementation may keep a temporary compatibility path.

## Requirements

1. The revised contract must replace the current `ensure-ready` / `wait-until-recovered` split with a `wait-until-ready` model that is consistent with the agreed constraints.

2. The revised contract must define command responsibilities under the two mutually exclusive selectors, `--project-path` and `--base-url`.

3. The revised contract must define `<project>/Temp/UnityPuerExec/session.json` as the project-local discovery artifact and make its minimum stable fields explicit.

4. The revised contract must state that `exec` may implicitly prepare Unity only in `--project-path` mode and that `get-result` never does so.

5. The revised contract must preserve `wait-for-log-pattern` as an observation-oriented command and make its no-target behavior explicit.

6. The revised contract must add `get-log-source` with selector-aware source reporting that does not over-promise stable server identity in `--base-url` mode.

7. The revised contract must add `ensure-stopped` with the already agreed asymmetry between project-scoped and base-url-scoped behavior.

8. The revised contract must remove `--keep-unity` from the formal surface and record the compatibility direction clearly enough for `T1.4.2` and `T1.4.3`.

9. The revised contract must keep machine-readable failure semantics for missing session discovery, stale discovery, selector conflict, and observation with no eligible target.

10. The revised contract must remain implementation-language-agnostic and should constrain later baseline work rather than perform it.

11. The revised contract must be validated against representative AI-agent usage scenarios rather than only by command-by-command prose inspection.

12. The revised contract must not preserve redundant commands or parameters unless a retained shortcut clearly reduces caller effort without adding a second semantic path.

    - direct-service readiness-only usage: the agent knows only `--base-url`, wants health confirmation without execution, and should be able to use the readiness model without triggering project discovery or Unity launch semantics
    - direct-service execution: the agent already knows `--base-url`, wants to run a script, and must not be forced into project discovery or Unity launch semantics
    - project-scoped observation: the agent waits for a log pattern from a known project and needs the contract to make the observable-target requirements and no-target outcome clear
    - log-source-first workflow: the agent asks `get-log-source` first to understand the current observable source before deciding whether to inspect logs directly or use `wait-for-log-pattern`
    - direct-service observation: the agent knows only `--base-url`, needs to observe logs or wait for a pattern, and the contract must describe the current best-effort observable source without implying stable server identity
    - explicit no-target observation failure: the agent invokes observation commands with no eligible observable target and the contract must return `no_observation_target`
    - project-scoped forced stop: the agent wants to guarantee the Unity project is stopped and must be able to distinguish inspect-only, timeout-then-kill, and immediate-kill modes
    - direct-service stop inspection: the agent knows only `--base-url` and must be able to inspect stop state without being offered an impossible kill path
    - async result follow-up: the agent receives a running job from `exec` and must use `get-result` without any implicit reprovisioning behavior
    - session-missing-versus-auto-prepare workflow: the agent compares project-scoped `exec` or `wait-until-ready`, which may auto-prepare, against non-launching commands such as `get-result` or observation commands, which must return machine-readable `session_missing` or `session_stale` instead of silently reprovisioning
    - redundant-surface challenge: the agent compares `wait-until-ready` against a project-scoped `exec` workflow and the revised contract must justify why the shortcut improves discovery without creating a second required semantic path
    - removed-parameter challenge: the agent completes readiness, execution, observation, and stop workflows without needing `--keep-unity`
  - the scenario review explicitly checks these formal parameter paths:
    - `--project-path` selector path
    - `--base-url` selector path
    - selector conflict when both are supplied
    - session discovery miss and stale-session handling
    - `ensure-stopped` inspect-only mode
    - `ensure-stopped` timeout-then-kill mode
    - `ensure-stopped` immediate-kill mode
    - `--base-url` stop inspection without kill capability

## Risks

- If `wait-until-ready` is described inconsistently as both a distinct command and only an alias, later implementation may drift.
- If the session artifact contract is underspecified beyond the minimum fields, later implementations may append incompatible semantics by accident.
- If `get-log-source` over-promises identity guarantees in `--base-url` mode, later implementations may be forced into premature session modeling.
- If the `ensure-stopped` asymmetry is not stated sharply enough, later code may accidentally allow destructive behavior in `--base-url` mode.
