# Plan: T1.4.1.1 Revise CLI Session Discovery And Lifecycle Contract

## Scope

Revise the formal CLI contract after `0007` so the session-discovery model and Unity lifecycle semantics are stable before `T1.4.2` starts moving implementation code.

This task should codify the already agreed direction that project-local discovery should become the normal path, that `ensure-ready` is the only command allowed to launch Unity, that `wait-until-recovered` and `wait-for-log-pattern` are observation-oriented commands against an existing session, and that `--keep-unity` should leave the formal surface.

This task should still decide the exact discovery artifact contract, selector precedence, compatibility policy, and machine-readable outcomes needed to make that direction executable.

This task should not implement the revised contract in code, move the CLI out of the skill-owned location, or decide the final packaging strategy.

## Requirements

1. The revised contract must make project-local discovery from the Unity project the normal session-addressing model for CLI commands.

2. The revised contract must treat an explicit `--base-url` as an override/debug path rather than the primary workflow.

3. The revised contract must treat `ensure-ready` as the only command that may launch Unity.

4. The revised contract must treat `wait-until-recovered` and `wait-for-log-pattern` as observation-oriented commands that do not launch Unity.

5. The revised contract must treat `ensure-ready` as keeping the launched Unity session alive by definition rather than relying on `--keep-unity`.

6. The revised contract must remove `--keep-unity` from the formal surface and replace it with explicit cleanup-ownership rules.

7. The revised contract must define a per-command addressing matrix for `ensure-ready`, `wait-until-recovered`, `wait-for-log-pattern`, `exec`, and `get-result`, including which inputs each command accepts, whether project-local discovery applies, whether `--base-url` is allowed, and which input wins on conflict.

8. The revised contract must define the exact project-local discovery artifact under `Temp/`, including path/name, minimum stable fields, writer, authority rules, refresh/removal lifecycle, and concurrency/cardinality expectations.

9. The revised contract must define validation and stale-session checks before the CLI trusts a discovered endpoint.

10. The revised contract must clarify the semantics of `wait-until-recovered` when no recovery event is needed and of `wait-for-log-pattern` when no eligible session or log observation target exists.

11. The revised contract must define machine-readable failure semantics for discovery miss, stale session discovery, conflicting address inputs, and observation-only commands with no eligible target whenever those states are part of the formal surface.

12. The revised contract must explicitly decide the compatibility policy for demoting `--base-url` and removing `--keep-unity`, including whether later implementation work keeps a transitional compatibility path.

13. The revised contract must explicitly re-audit any affected workflow, output, and help sections in `0007`, not only the parameter list.

14. The revised contract must remain implementation-language-agnostic and should constrain `T1.4.2` rather than perform it.

   - Write scope: plan notes and later durable decision update only.
   - Parallelism: no.

2. Define the preferred session discovery model.
   - Goal: Lock in project-local discovery as the primary model and define the remaining override/debug role of explicit endpoint input.
   - Write scope: durable decision update only.
   - Parallelism: limited.

3. Define the per-command addressing matrix.
   - Goal: Decide, command by command, whether project-local discovery applies, whether `--base-url` remains accepted, and which input wins when more than one address source is supplied.
   - Write scope: durable decision update only.
   - Parallelism: limited; depends on the session-discovery decision.

4. Define the session file contract if project-local discovery is adopted.
   - Goal: Decide the stable file location, minimum fields, owner, authority rules, cardinality, and stale-session checks.
   - Write scope: durable decision update only.
   - Parallelism: limited.

5. Redefine command lifecycle responsibilities.
   - Goal: Codify that only `ensure-ready` may launch Unity, that `wait-until-recovered` and `wait-for-log-pattern` only attach or observe, and that `ensure-ready` keeps Unity alive by definition.
   - Write scope: durable decision update only.
   - Parallelism: limited; depends on the earlier decisions.

6. Define failure and cleanup semantics affected by the revision.
   - Goal: Decide machine-readable outcomes for discovery miss, stale session rejection, address conflicts, no-observable-target cases, and the cleanup contract after removing `--keep-unity` from the formal surface.
   - Write scope: durable decision update only.
   - Parallelism: limited; depends on the earlier decisions.

7. Distill the revision into durable contract documentation.
   - Goal: Update `docs/decisions/0007-formal-cli-contract.md` so later baseline work has one authoritative CLI contract.
   - Write scope: `docs/decisions/0007-formal-cli-contract.md`, plus roadmap/status pointers if task state changes.
   - Parallelism: no.

## Validation

- Validation owner: main agent, with one clean-context subagent review before human review.
- Direct checks:
  - the revised contract locks project-local discovery as the normal session discovery path
  - the revised contract contains a command-by-command addressing matrix
  - the role of `--base-url` is explicit as an override/debug path after the revision
  - the revised contract makes session-file ownership, authority, and cardinality explicit if project-local discovery is adopted
  - command-by-command Unity launch responsibility is explicit
  - the revised contract makes `ensure-ready` the only launch-capable command
  - the revised contract makes `wait-until-recovered` and `wait-for-log-pattern` observation-only
  - the formal removal or compatibility treatment of `--keep-unity` is explicit
  - discovery miss, stale discovery, and address-conflict behavior are explicit when they remain part of the formal surface
  - no-observable-target behavior is explicit for observation-only commands
  - the revised `0007` stays internally consistent across workflow, output, and help sections rather than changing only parameter wording
  - the revised contract gives `T1.4.2` one authoritative target instead of leaving both old and new lifecycle models in force
- Deferred checks:
  - implementation of any session file, endpoint probing, or help-text changes

## Risks

- If the revision keeps both project-local discovery and explicit endpoint input equally primary, later help and implementation may remain confusing.
- If command lifecycle boundaries stay vague, `T1.4.2` may preserve surprising launch side effects.
- If the session file contract is underspecified, later implementations may write incompatible discovery artifacts.
- If the revision leaves failure semantics generic, later code may silently reintroduce contract decisions through ad hoc exceptions.
