# Plan: T1.4.4 Formal Help Surface

## Plan Type

Implementation Plan

## Scope

Implement the formal `unity-puer-exec` help surface as a stable, agent-facing product contract.

This task covers top-level help, per-command help, deep help entry points, workflow examples, and help-structure tests inside `unity-puer-exec`.

This task does not rewrite repository-facing usage docs outside the CLI help surface. That follow-up remains in `T1.4.5`.

## Agreed Constraints

- `T1.4.4` in `unity-puer-exec` is limited to the formal CLI help surface; repository doc rewrites stay in `T1.4.5`.
- The help surface is a product contract and should act as a skill replacement for normal CLI discovery.
- The help contract should optimize for progressive disclosure and token efficiency.
- The command inventory in scope for this task is exactly:
  - `wait-until-ready`
  - `wait-for-log-pattern`
  - `get-log-source`
  - `exec`
  - `get-result`
  - `ensure-stopped`
- Top-level `--help` is the discovery entry point, not the place to inline all detailed help.
- Every command must support default `--help`.
- Every command must support `--help-args`.
- Commands may expose optional deep help entry points, but the currently agreed set for this task is `--help-status` on all existing commands.
- Deep help output should show only the requested layer and should not repeat default help content.
- Top-level help should own workflow examples through `--help-example <example-id>`; child commands should not implement `--help-examples`.
- `unity_puer_exec.py` should remain the CLI entry and execution router, while help metadata and rendering should move into a small dedicated helper module instead of further inflating the entry file.
- The accepted help entry forms for this task are:
  - `unity-puer-exec --help`
  - `unity-puer-exec --help-example <example-id>`
  - `unity-puer-exec <command> --help`
  - `unity-puer-exec <command> --help-args`
  - `unity-puer-exec <command> --help-status`
- `--help-example` is top-level only; it is not accepted after a command token.
- `--help-args` and `--help-status` are command-level only; they are not accepted at top level.
- Help requests short-circuit normal command execution and bypass required runtime argument validation for that target help page.
- For deterministic parsing, a deep help invocation should contain only the command token plus the requested help flag; extra execution arguments should be treated as usage errors rather than ignored.
- Top-level help must include `Overview`, `Commands`, `Global Selector Rules`, and `Common Workflows`.
- Top-level command entries should include one-line purpose/positioning text and explicitly point to `<command> --help`.
- `Global Selector Rules` should remain short but must call out the exclusivity and typical use of `--project-path` and `--base-url`.
- Per-command default help must include `Quick Start`, `More Help`, and `Related Workflows`.
- `Quick Start` uses a stable shape: one-sentence purpose, one or two common invocations, and an optional shortcut/equivalence note when needed.
- Default command examples should prefer `--project-path` where that command supports it.
- Per-command `More Help` should list deep help flags in fixed order.
- Per-command `Related Workflows` should list only workflow examples where the command is a main actor or key transition point.
- `--help-args` uses a shared ordered template with `Arguments`, `Selector Rules`, and `Timeout Rules`; inapplicable sections are omitted rather than rendered as `none`.
- `--help-status` uses a shared template with `Success Statuses` and `Non-success Statuses`; success states share the rule that their exit code is `0`, while each non-success state records status name, exit code, and one-line branching meaning.
- Workflow examples should be stable enough for discovery and testing, but example IDs may change when the recommended usage materially changes.
- Example IDs should use meaningful task-oriented names that are easy to infer from the ID itself.
- Initial workflow examples for this task are:
  - `cold-start-exec-and-get-result`
  - `long-job-and-log-pattern`
  - `request-editor-exit-via-exec`
- `cold-start-exec-and-get-result` assumes Unity is not running, script input comes from a file, and the first `exec` response is `running`.
- `long-job-and-log-pattern` assumes Unity is already running; it first establishes observation with a short/non-blocking `wait-for-log-pattern`, then starts a fake long-running workload, then blocks on `wait-for-log-pattern` to avoid missing an early log line.
- `request-editor-exit-via-exec` assumes Unity is already available and demonstrates a script-based normal exit request.
- `wait-until-ready` remains discoverable but intentionally de-emphasized; it does not need its own workflow example in this task.
- `ensure-stopped` should explain its stopped-state contract in command help, but it should not be presented as the normal exit workflow.
- `get-log-source` does not need an initial workflow example in this task; if a fourth example becomes necessary later, it is the first candidate.
- Unknown `--help-example` IDs should return exit code `2`, write a text usage error to stderr, keep stdout empty, and list available example IDs.
- Workflow examples may use tiny illustrative script snippets or placeholder file names, but they must not depend on hidden helper assets, undocumented repo-local files, or behavior outside the formal CLI contract.
- Workflow examples should describe only the command sequence, expected observable outputs, and contract-level interpretation needed to follow the workflow; they should not imply extra guarantees beyond the CLI contract.
- `long-job-and-log-pattern` should explicitly present its workload as a fake demonstration workload rather than a built-in compile command or productized operation.
- Help structure should receive direct structural tests in this task; agent/subagent usability review can remain a later validation layer.

## Requirements

- Replace the current argparse-default help surface with a formal help contract that supports progressive discovery.
- Keep top-level help short enough to act as a navigation page while still exposing the command tree, selector model, and workflow entry points.
- Ensure each command has a default help page that lets an agent invoke the common path without reading deep help first.
- Ensure each command has argument detail help sufficient to resolve selectors, required inputs, and timeout semantics without consulting repo docs.
- Ensure each command has status help sufficient to understand machine states and non-zero exits without reading source code.
- Ensure workflow examples demonstrate the recommended multi-command paths rather than isolated command snippets.
- Keep examples human-meaningful and agent-useful without assuming any specific agent shell timeout model.
- Preserve current behavioral command contract from `T1.4.3`; this task changes help discoverability, not non-help command semantics.

## Work Items

1. Implement the agreed formal help routing and data model
   - Goal: encode the already-settled top-level help, per-command help, deep help flags, and workflow example routing without relying on argparse's default output shape.
   - Write scope: `cli/python/unity_puer_exec.py` for routing and `cli/python/` help helper module(s) for metadata/rendering.
   - Parallelism: should complete before detailed help text authoring and test updates.

2. Implement top-level help and workflow example surfaces
   - Goal: render the stable top-level sections, attach command discovery text, expose `--help-example <example-id>`, and handle unknown example IDs with a useful non-zero result.
   - Write scope: `cli/python/unity_puer_exec.py` for dispatch and help helper module(s) for content/rendering.
   - Parallelism: depends on Work Item 1.

3. Implement per-command default help pages
   - Goal: render `Quick Start`, `More Help`, and `Related Workflows` for all current commands with stable structure and agreed emphasis.
   - Write scope: help helper module(s) under `cli/python/`, plus `unity_puer_exec.py` wiring.
   - Parallelism: depends on Work Item 1; can overlap with Work Item 4 once the shared rendering shape is established.

4. Implement shared deep help pages
   - Goal: render `--help-args` and `--help-status` for each existing command using stable shared templates with command-specific content.
   - Write scope: help helper module(s) under `cli/python/`, plus `unity_puer_exec.py` wiring.
   - Parallelism: depends on Work Item 1; can overlap with Work Item 3 after the command metadata shape is fixed.

5. Author the initial workflow examples
   - Goal: encode the three agreed examples with `Goal`, `Steps`, and `What To Notice`, including expected observations and contract-level interpretation.
   - Write scope: help helper module(s) under `cli/python/`.
   - Parallelism: depends on Work Items 1 and 2.

6. Add structural help tests
   - Goal: verify the new help surfaces, example routing, and unknown-example failure path without overfitting tests to full prose bodies.
   - Write scope: `tests/test_unity_session_cli.py` and any new test module if separation improves clarity.
   - Parallelism: depends on Work Items 2 through 5.

## Out Of Scope

- Rewriting repository-facing docs, README content, or governance docs beyond the roadmap plan pointer needed to activate this plan.
- Changing non-help command behavior, HTTP payload behavior, or exit-code semantics already established by `T1.4.3`.
- Adding a `get-log-source` workflow example unless implementation evidence shows the initial three examples are insufficient for current command discovery.
- Adding subagent usability automation in this task.

## Integration

Single-repo integration in `unity-puer-exec`; no cross-repo merge point is needed.

## Validation

Validation owner: implementer.

Direct checks:

- confirm `docs/roadmap.md` points `T1.4.4` at this active plan before implementation starts
- confirm top-level `--help` renders the agreed section structure and includes the workflow example IDs
- confirm each in-scope command (`wait-until-ready`, `wait-for-log-pattern`, `get-log-source`, `exec`, `get-result`, `ensure-stopped`) renders default `--help`, `--help-args`, and `--help-status`
- confirm child commands do not expose a separate `--help-examples` surface
- confirm `--help-example <example-id>` renders each known example
- confirm an unknown example ID returns exit code `2`, writes to stderr, leaves stdout empty, and lists available example IDs
- confirm deep help invocations reject extra execution arguments instead of silently ignoring them
- confirm structural tests cover top-level help, child help entry points, example routing, and unknown-example failure

Deferred checks:

- later subagent-driven usability validation can test whether an agent can self-discover the recommended workflows from the new help surface without repo-specific skill guidance

## Risks

- Help text may become too verbose and lose the intended progressive-disclosure benefit.
- A partially custom help router may diverge from argparse parsing behavior if flag handling is not wired carefully.
- Stable section structure may still drift if tests only cover a narrow subset of the contract.
- Workflow examples may accidentally imply behavior not guaranteed by the actual CLI contract if wording is not kept precise.

## Exit Criteria

- `T1.4.4` has an approved executable plan that captures the agreed help contract without requiring chat-history replay.
- `docs/roadmap.md` points `T1.4.4` at this plan file.
- The plan is specific enough that implementation can proceed without reopening the settled help-structure decisions covered in `Agreed Constraints`.
