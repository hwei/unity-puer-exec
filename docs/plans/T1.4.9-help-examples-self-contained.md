# Plan: T1.4.9 Help Examples Self-Contained

## Plan Type

Implementation Plan

## Scope

Make the explicit exit-request workflow example self-contained so an agent can execute it without guessing hidden script contents. This task updates the help-surface example contract for `request-editor-exit-via-exec`, its rendered output, and the structural tests that verify that contract. It does not change command semantics, the top-level empty-invocation contract, the `wait-until-ready` ownership model, or attempt to canonize currently non-validated workflow snippets for the other examples.

## Agreed Constraints

- `T1.4.9` only changes help examples and their tests.
- This task only requires an inline literal script body for the `request-editor-exit-via-exec` example.
- The current example structure stays top-level only through `unity-puer-exec --help-example <example-id>`.
- Example IDs remain:
  - `cold-start-exec-and-get-result`
  - `long-job-and-log-pattern`
  - `request-editor-exit-via-exec`
- Example output remains structured as `Goal`, `Steps`, and `What To Notice`.
- Within `request-editor-exit-via-exec`, the script step uses a fixed sub-shape:
  - command line
  - `Example script body:`
  - indented literal JavaScript snippet
  - `Expected observation:`
- The inline `request-exit.js` snippet is fixed as:
  - `const EditorApplication = puer.loadType('UnityEditor.EditorApplication');`
  - `EditorApplication.Exit(0);`
- `What To Notice` stays contract-focused; it should not grow into general design rationale.
- This task does not absorb `T1.4.7` or `T1.4.8`.
- The other workflow examples remain illustrative and may continue to reference placeholder script paths until a later task can validate runnable snippets against a complete tool loop.

## Requirements

- `unity-puer-exec --help-example <example-id>` must expose enough information to execute the documented workflow without relying on unpublished script contents.
- `request-editor-exit-via-exec` must render an `Example script body:` block directly under its `--file X:/scripts/request-exit.js` step before `Expected observation:`.
- The inline exit-request snippet must be a literal JavaScript body using `puer.loadType('UnityEditor.EditorApplication')` followed by `EditorApplication.Exit(0);`.
- The rendered example must not require any hidden repository file contents or unstated setup beyond Unity already being available for the selected project.
- The rendered example format preserves the current `Goal` / `Steps` / `What To Notice` section structure and keeps the step-internal ordering stable for tests.
- Structural tests must verify the `Example script body:` marker and the literal exit-request snippet so later edits cannot silently regress to placeholder-only guidance for this example.

## Risks

- Over-explaining examples could bloat help output and undermine the token-saving design from `T1.4.4`.
- Even a tiny exit-request snippet could drift from real host expectations if the host-side script contract changes later.
- Leaving the other workflow examples illustrative means the overall help surface still has partially placeholder-driven examples until follow-up work lands.

## Exit Criteria

- `request-editor-exit-via-exec` no longer requires the reader to infer what `request-exit.js` contains.
- The rendered exit workflow includes the exact inline `EditorApplication.Exit(0)` snippet in a stable `Example script body:` block.
- Tests fail if a future edit regresses that exit example back to placeholder-only script guidance.
