# Plan: T2.2 Documentation Structure For Minimal Agent Token Cost

## Plan Type

Implementation Plan

## Scope

Reorganize repository-facing workflow and planning documentation so an agent can usually load only the minimum document slice needed for the current workflow state.

This task covers documentation structure, document boundaries, and document entry guidance.

This task does not change the underlying workflow semantics, task-state model, review order, or long-lived decision content unless a structural rewrite is required to preserve the same behavior in a lower-token form.

## Agreed Constraints

- The new documentation structure must preserve the current workflow semantics.
- This task optimizes token cost and lookup shape, not repository process policy.
- The current workflow sequence, planning model, retrospective handoff model, and roadmap meaning should remain materially the same after the rewrite.
- Agents should be able to discover the next needed document without loading large unrelated rule sections by default.
- Stable background information should become more pointer-oriented and more demand-loaded.
- The output should remain usable by both fresh-session agents and human readers.
- `AGENTS.md` remains the repo-level entry surface because the execution environment already treats it as the top-level instruction file.
- The implementation may add a tiny docs-local entry file, but `AGENTS.md` must remain the canonical external dispatcher into `docs/`.
- `docs/roadmap.md` remains the canonical task and dependency source of truth even if a smaller active-view document is introduced.
- Any newly introduced summary or active-view file must point back to one authoritative source instead of becoming a competing source of truth.

## Shared Context

- Task anchor and status live in `docs/roadmap.md` under `T2.2`.
- Current workflow rules live in `docs/workflow.md`.
- Current planning rules live in `docs/planning.md`.
- Current active-state summary lives in `docs/status.md`.
- The current roadmap mixes active work, hierarchy rules, and completed-task references in one file: `docs/roadmap.md`.

## Requirements

- Define a target documentation structure that separates:
  - entry guidance
  - current-status guidance
  - workflow-sequence guidance
  - plan-authoring guidance
  - active roadmap state
  - durable decisions
- Minimize the amount of documentation an agent must read before taking the next correct action in common workflow states.
- Preserve deterministic execution by keeping authoritative rules discoverable and unambiguous.
- Avoid duplicating the same rules across multiple documents merely to create smaller files.
- Make default read paths explicit for at least these states:
  - fresh-session orientation
  - continuing active work
  - writing or revising a plan
  - implementing an approved plan
  - retrospective disposition and plan deletion
  - closing out a completed task with no retrospective findings
- Define what content should move, what should stay, and what should be replaced by pointers.
- Define how all existing inbound references to moved material will be preserved or updated atomically.

## Work Items

1. Analyze the current document graph and default read cost
   - Goal: identify where agents currently load unrelated information and which files mix multiple responsibilities.
   - Write scope: planning notes in this plan; later changes may touch `docs/workflow.md`, `docs/planning.md`, `docs/status.md`, `docs/roadmap.md`, and new lightweight docs.
   - Parallelism: can run before detailed structure drafting; no code-write parallelism needed.

2. Design a token-minimized target structure
   - Goal: specify the smallest set of entry documents and specialized rule documents that preserves current semantics.
   - Write scope: this plan and any human-review discussion output; no implementation changes yet.
   - Parallelism: depends on Work Item 1.

3. Define migration rules for each existing document
   - Goal: map current sections to their target homes and identify which files should become indexes, which should stay authoritative, and which should become active views.
   - Write scope: this plan and subsequent implementation changes to docs.
   - Parallelism: depends on Work Item 2.

4. Define validation for the new structure
   - Goal: describe how to verify that agents can orient, plan, execute, and close tasks with lower default token cost and without losing required rules.
   - Write scope: this plan and later verification notes.
   - Parallelism: depends on Work Items 2 and 3.

5. Define an explicit pointer-migration checklist
   - Goal: ensure every existing hard-coded pointer into workflow, planning, roadmap, and entry docs is either preserved or updated in the same change.
   - Write scope: this plan and later implementation edits to `AGENTS.md` and affected docs.
   - Parallelism: depends on Work Items 2 and 3.

## Open Questions

- Should workflow and planning docs be split into more files, or should they stay single-file with explicit quick-read and deep-read sections?

## Out Of Scope

- Changing task IDs, task states, or dependency semantics.
- Changing CLI or package product requirements.
- Revising existing durable decisions for product behavior.
- Rewriting plans, decisions, or roadmap items unrelated to documentation structure.
- Building automation that enforces document read paths.

## Integration

Single-owner docs integration in `unity-puer-exec`; no cross-repo merge point is needed.

## Validation

Validation owner: implementer.

Direct checks:

- confirm the target structure defines a minimal default read path for each common workflow state
- confirm the target structure defines an explicit minimal read path for retrospective disposition and plan deletion
- confirm the new structure preserves the current workflow and planning semantics
- confirm `AGENTS.md` still acts as the repo-level dispatcher and that any docs-local entry file remains subordinate to it
- confirm `docs/roadmap.md` still acts as the canonical task and dependency source of truth
- confirm each moved rule still has exactly one authoritative home
- confirm entry documents prefer pointers over repeated prose
- confirm the implementation includes an explicit migration table or checklist for inbound references to moved content

Deferred checks:

- a later fresh-session execution task can test whether another agent reaches the correct next document with fewer loaded tokens

## Risks

- Over-splitting may reduce token cost but make rule discovery brittle.
- Under-splitting may preserve today’s mixed-responsibility files and miss most of the benefit.
- Rephrasing during structural moves may accidentally change workflow semantics.
- Multiple “summary” documents may drift if authority boundaries are not explicit.

## Exit Criteria

- The repository has a reviewed plan for restructuring documentation around minimal default read cost.
- The plan identifies the target document set, the role of each document, and the intended read path for common workflow states.
- The plan explicitly preserves current workflow semantics while reducing unnecessary default reads.
- The plan is specific enough for implementation without requiring the executor to infer undocumented structure decisions from chat history.
