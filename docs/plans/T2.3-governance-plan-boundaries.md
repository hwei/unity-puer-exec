# Plan: T2.3 Clarify Governance-Plan Boundaries

## Plan Type

Decision Plan

This plan uses the current repository term so it remains readable under the existing workflow rules. The task itself is to replace that term with `Governance Plan` and tighten its boundary.

## Scope

Revise workflow and planning documentation so the repository uses `Governance Plan` instead of `Decision Plan`, and so the plan-type boundary follows the agreed distinction between final-product work and project-management-document work.

This task covers governance-facing rule changes in repository process documents.

This task does not directly reorganize the broader documentation set for token minimization. That later work stays in `T2.4`.

## Agreed Constraints

- This task should be framed and executed as governance-document work, not as a precursor to a separate implementation task.
- The replacement of `Decision Plan` with `Governance Plan` is itself part of this task's direct output.
- `Governance Plan` applies when the main output is project-management documentation rather than final product artifacts.
- `Implementation Plan` applies when the main output is code, tests, or product-user-facing documentation.
- `ReadMe.md`, CLI `--help`, package usage docs, and other user-facing contract docs count as product-user-facing documentation.
- `AGENTS.md`, `docs/workflow.md`, `docs/planning.md`, `docs/roadmap.md`, `docs/status.md`, and `docs/decisions/` count as project-management documentation.
- The `docs/decisions/` directory name remains unchanged for now; this task clarifies plan-type terminology, not the archival naming of existing governance-document locations.
- A `Governance Plan` may directly modify long-lived governance documents without requiring a follow-up implementation task.
- If a primarily governance task includes auxiliary changes that would directly alter final product behavior, test behavior, or product-user-facing documentation, those changes should be split into an `Implementation Plan`.
- If a governance task is too large or unstable for one task, any split follow-up should default to another `Governance Plan` unless its main output shifts to final product artifacts.
- The rewrite should minimize new ambiguity and should prefer simple classification rules over abstract conceptual distinctions.

## Shared Context

- Task anchor and status live in `docs/roadmap.md` under `T2.3`.
- Current workflow rules live in `docs/workflow.md`.
- Current planning rules and plan-type definitions live in `docs/planning.md`.
- The plan template lives in `docs/plan-template.md`.
- `T2.2` is currently `dropped` in `docs/roadmap.md`.
- `T2.4` is the downstream consumer of this clarification.
- `T2.5` records the follow-up that `ReadMe.md` should be user-facing while developer guidance points to `AGENTS.md`.

## Requirements

- Replace `Decision Plan` with `Governance Plan` in repository process documents.
- Make the primary classification rule explicit: classify by the task's main output, especially whether it directly changes final product artifacts or project-management artifacts.
- State explicitly that product-user-facing documentation belongs with `Implementation Plan`, not `Governance Plan`.
- State explicitly that governance-facing documentation may be modified directly as part of a `Governance Plan`.
- State explicitly when auxiliary work inside a governance task must be split into an `Implementation Plan`.
- Preserve the existing review order and overall workflow semantics except where the old `Decision Plan` naming and boundary caused ambiguity.
- Leave `Exploration Plan` semantics unchanged unless a minimal wording adjustment is required for consistency.

## Work Items

1. Define the authoritative plan-type boundary
   - Goal: turn the agreed classification rule into clear workflow and planning language.
   - Write scope: `docs/planning.md`, `docs/workflow.md`, and this plan.
   - Parallelism: should complete before any template rewrite.

2. Rename `Decision Plan` to `Governance Plan`
   - Goal: update process documents so the new term is consistent and not half-migrated.
   - Write scope: `docs/planning.md`, `docs/plan-template.md`, `docs/workflow.md`, `docs/roadmap.md`, and `AGENTS.md` only if its wording needs a plan-type name update.
   - Parallelism: depends on Work Item 1.

3. Clarify direct-execution rules for governance work
   - Goal: state that governance plans may directly modify long-lived governance documents and only split into implementation work when the main output shifts to final product artifacts.
   - Write scope: `docs/planning.md`, `docs/workflow.md`, and possibly `docs/plan-template.md`.
   - Parallelism: depends on Work Items 1 and 2.

4. Align examples and templates with the new boundary
   - Goal: make the plan template and examples reinforce the correct classification instead of reintroducing the old ambiguity.
   - Write scope: `docs/planning.md`, `docs/plan-template.md`, and any short examples needed in `docs/workflow.md`.
   - Parallelism: depends on Work Items 1 through 3.

## Out Of Scope

- Executing `T2.4` token-minimization document restructuring.
- Rewriting `ReadMe.md` itself.
- Changing product behavior, code, tests, or user-facing contracts.
- Broadly renaming historical task descriptions unless needed for current rule clarity.
- Renaming the `docs/decisions/` directory.

## Integration

Single-owner governance-doc update in `unity-puer-exec`; no cross-repo merge point is needed.

Deferred checks:

## Risks

- Leaving mixed old and new terminology behind would recreate ambiguity.
- Overwriting the old term without enough classification examples could still leave edge cases unclear.
- Pushing too much scope into this task could turn a governance clarification into an unrelated documentation overhaul.

## Exit Criteria

- Repository process docs replace `Decision Plan` with `Governance Plan`.
- The rules clearly distinguish governance-document work from final-product work.
- The rules explicitly allow governance plans to directly modify long-lived governance documents.
- The rules explicitly require splitting out auxiliary work when that work's main output changes final product artifacts.
- The resulting documentation is specific enough that `T2.4` can be planned without repeating the same plan-type confusion.
