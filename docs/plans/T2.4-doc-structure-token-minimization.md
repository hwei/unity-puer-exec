# Plan: T2.4 Documentation Structure For Minimal Agent Token Cost

## Plan Type

Governance Plan

## Scope

Reorganize repository-facing governance documentation so an agent can usually load only the minimum document slice needed for the current workflow state.

This task covers governance-document entry paths, authority boundaries, and section/file placement inside `AGENTS.md` and `docs/`.

This task does not change workflow semantics, roadmap semantics, plan-type semantics, task-state meanings, or final product behavior.

## Agreed Constraints

- This task optimizes token cost and default lookup shape, not repository process policy.
- The required workflow sequence must remain materially unchanged.
- The planning model, retrospective handoff model, and roadmap task semantics must remain materially unchanged.
- `AGENTS.md` remains the repo-level external entry surface.
- `docs/roadmap.md` remains the single canonical task and dependency source of truth.
- This task must not split roadmap authority across `active`, `archive`, or competing summary files.
- `docs/index.md` should be introduced as the only docs-local router file, and it must remain subordinate to `AGENTS.md`.
- Each governance rule must have exactly one authoritative home after the rewrite.
- Entry and router documents should prefer pointers over repeated prose.
- The rewrite must cover these workflow states explicitly:
  - fresh-session orientation
  - continuing active work
  - writing or revising a plan
  - implementing an approved plan
  - retrospective disposition and plan deletion
  - closing out a completed task with no retrospective findings
- The target structure should reduce default reads by moving specialized closeout and planning detail out of broad entry documents.
- This task may directly modify governance documents but must not directly modify final product artifacts.

## Shared Context

- Task anchor and status live in `docs/roadmap.md` under `T2.4`.
- Repo-level dispatcher rules live in `AGENTS.md`.
- Current workflow rules live in `docs/workflow.md`.
- Current planning rules live in `docs/planning.md`.
- Current active-state summary lives in `docs/status.md`.
- Current roadmap authority lives in `docs/roadmap.md`.
- The dropped precursor plan is `docs/plans/T2.2-doc-structure-token-minimization.md`; reuse only its stable constraints, not its old plan-type framing.

## Requirements

- Define a minimal read path for each common workflow state without requiring the agent to load unrelated governance detail by default.
- Keep `AGENTS.md` as the only repo-level external dispatcher, and keep `docs/index.md` short enough to function as a router rather than a rule dump.
- Preserve `docs/roadmap.md` as the only authoritative task graph while reducing how often an agent must open unrelated planning prose before reaching the active task context.
- Separate broad workflow sequence guidance from specialized closeout guidance so implementation-oriented reads do not load retrospective and distillation detail by default.
- Separate planning entry guidance from deeper planning rules so plan authors can reach plan-type selection, agreed-constraint expectations, and the template without always loading the full planning rationale.
- Keep `docs/status.md` focused on current focus, blockers, and next steps rather than duplicating workflow or planning rules.
- Define the target role of each governance document that remains after the rewrite.
- Define an explicit migration checklist for inbound pointers from `AGENTS.md` and existing `docs/*` files.
- Keep renamed or newly introduced files semantically narrow enough that an agent can infer whether to open them from the workflow state alone.

## Target Structure

The rewrite should converge on this governance-document set and role split:

- `AGENTS.md`
  - role: repo-level external dispatcher, environment setup, repo boundary, and top-level pointers into `docs/`
  - must stay short and must not become a second workflow or planning rule dump
- `docs/index.md`
  - role: docs-local router keyed by workflow state
  - should point readers to the minimum next document instead of restating detailed rules
- `docs/status.md`
  - role: current focus, blockers, and next steps only
  - should not carry planning or workflow rules beyond a tiny pointer back to `docs/index.md`
- `docs/roadmap.md`
  - role: single canonical task graph, task-state rules, dependencies, and plan/output pointers
  - must remain the only authoritative roadmap source
- `docs/workflow.md`
  - role: required sequence plus a compact read-path map for execution states
  - should point to specialized planning and closeout docs instead of carrying all details inline
- `docs/workflow-closeout.md`
  - role: distillation, retrospective, disposition, and plan-deletion rules
  - should hold the specialized closeout detail currently mixed into `docs/workflow.md`
- `docs/planning.md`
  - role: planning quickstart covering plan-type selection, agreed-constraint expectations, minimum plan shape, review order, and the template pointer
  - should stay small enough to serve as the first planning read
- `docs/planning-rules.md`
  - role: deeper planning rationale and subagent-friendly authoring rules that are not required for every planning read
- `docs/plan-template.md`
  - role: canonical plan skeleton only
- `docs/decisions/`
  - role: durable active decisions still in force

No `roadmap-active.md`, `roadmap-archive.md`, or competing summary roadmap file should be introduced.

## First-Read Paths

The rewritten docs should support these default first-read paths:

- fresh-session orientation
  - `AGENTS.md`
  - `docs/index.md`
  - `docs/status.md`
- continuing active work
  - `docs/status.md`
  - `docs/roadmap.md`
  - active plan file when one exists
- writing or revising a plan
  - `docs/workflow.md`
  - `docs/planning.md`
  - `docs/plan-template.md`
  - `docs/planning-rules.md` only when the quickstart pointer says more detail is needed
- implementing an approved plan
  - `docs/workflow.md`
  - `docs/roadmap.md`
  - active plan file
- retrospective disposition and plan deletion
  - `docs/workflow-closeout.md`
  - active plan file
  - `docs/roadmap.md` only if a follow-up task or disposition output must be recorded
- closing out a completed task with no retrospective findings
  - `docs/workflow-closeout.md`
  - active plan file

## Section Moves

Current material should move as follows:

- `docs/workflow.md`
  - keep: required sequence, plan rules summary, roadmap-maintenance summary, validation rules summary, truth hierarchy
  - move out: detailed distillation and retrospective rules into `docs/workflow-closeout.md`
- `docs/planning.md`
  - keep: plan-type selection, plan-file naming, agreed-constraint rule, minimum plan shape, quick pointer to template
  - move out: OpenSpec-inspired rationale and detailed subagent-friendly authoring rules into `docs/planning-rules.md`
- `docs/status.md`
  - keep only current focus, in-progress, blocked, and next
  - remove any future duplication of workflow or planning rule content
- `AGENTS.md`
  - add only the minimum pointer to `docs/index.md` and the canonical roles of `workflow`, `roadmap`, `status`, `plans`, and `decisions`
  - do not duplicate workflow, planning, or closeout rules there

## Migration Checklist

- Update `AGENTS.md` so its docs pointers match the new structure in the same change that adds or renames docs files.
- Add `docs/index.md` as the docs-local router in the same change that repoints `AGENTS.md`.
- Update `docs/workflow.md` and create `docs/workflow-closeout.md` atomically so closeout rules always have one live authoritative home.
- Update `docs/planning.md` and create `docs/planning-rules.md` atomically so planning pointers never reference missing detail.
- Preserve or update inbound references that currently point at moved workflow and planning sections.
- Leave `docs/roadmap.md` authoritative and avoid introducing any alternate roadmap summary file.

- Changing governance-plan, implementation-plan, or exploration-plan semantics.

## Validation

Validation owner: implementer.

Direct checks:

- confirm the plan fixes the target governance-document set instead of leaving file-splitting choices to implementation-time inference
- confirm the plan defines a minimal first-read path for each required workflow state
- confirm `AGENTS.md` remains the repo-level dispatcher
- confirm `docs/roadmap.md` remains the single canonical task and dependency source of truth
- confirm each moved rule or section has one authoritative home
- confirm the target structure keeps `docs/status.md` as a current-state page rather than a mixed rule document
- confirm the target structure explicitly covers retrospective disposition and no-retrospective closeout
- confirm the plan includes an explicit pointer-migration checklist

Deferred checks:

- fresh-session execution can later validate whether another agent reaches the right document path with fewer loaded tokens in practice

## Risks

- Over-splitting can reduce token cost while making rule discovery brittle.
- Under-splitting can preserve mixed-responsibility files and miss most of the benefit.
- Structural moves may accidentally change semantics if section authority is not explicit.
- A new docs-local router could become a second rule dump if it is not kept narrow.

## Exit Criteria

- The repository has a reviewed governance plan that fixes the target governance-document structure for lower default token cost.
- The plan identifies the role of each target governance document and the intended first-read path for each common workflow state.
- The plan preserves current workflow and planning semantics while reducing unnecessary default reads.
- The plan is specific enough that implementation does not need to infer additional document-structure decisions from chat history.
