## Context

The `exec` command currently wraps user JS code in an IIFE and evaluates it via `jsEnv.Eval()` (V8 script context). Static `import` statements are explicitly rejected in `UnityPuerExecProtocol.cs` because script context does not support them. To allow users to compose exec scripts from external JS modules, execution must move to ES module context.

PuerTS 3.0 exposes `JsEnv.ExecuteModule(specifier)` which delegates entirely to a configurable `ILoader`. The host validation project confirms that arbitrary strings are valid specifiers (`ExecuteModule("BOOT")` is used there), meaning our custom virtual scheme `puer-exec://` will pass through to the ILoader without path normalization.

## Goals / Non-Goals

**Goals:**
- Support static `import` declarations at the top of exec entry scripts
- Support relative filesystem imports resolved from the entry file's directory
- Support `--import-base-url` as an explicit filesystem or HTTP base for import resolution
- Support synchronous HTTP module fetching (for dev-server workflows)
- Add `--reset-jsenv-before-exec` to clear the JS module cache without restarting Unity
- Keep all existing exec behavior intact for scripts that do not use `import`

**Non-Goals:**
- Bundling or pre-fetching of imported modules by the CLI
- Package management
- Async entry functions or Promise-returning exec (blocked by existing protocol contract)
- CommonJS `require()` support

## Decisions

### Decision 1: Dual virtual module scheme (harness + entry)

The bridge ceremony (CompleteJob/FailJob wiring, ctx construction, result serialization, async detection) currently lives in the IIFE wrapper generated by `TryBuildWrappedScript`. In module mode, the wrapper cannot be inlined because static imports must appear at the module's top level, before any executable code.

**Solution:** Generate two virtual modules per request:

- `puer-exec://harness/<requestId>` — contains the bridge ceremony as top-level module code; imports the user's entry module via its resolved specifier.
- `puer-exec://entry/<requestId>` — present only when the entry code is a string (stdin/code mode) and the base URL is an HTTP scheme; allows the ILoader to serve the code string as if it were an HTTP URL so that relative imports from the entry are anchored correctly.

For `--file` mode the harness imports the real file path directly, making `puer-exec://entry/<id>` unnecessary. For stdin/code with a filesystem base URL the code is written to a temporary file in the base directory (so relative imports anchor there), and the harness imports that temp file path.

**Alternatives considered:**
- Write harness to a temp file and `ExecuteModule(path)`: works but introduces disk I/O and cleanup responsibility for the harness on every request. Virtual specifiers are cleaner.
- Inline all imports via C#-side bundling: explicitly out of scope per product direction.

### Decision 2: ILoader resolution priority

The custom `ILoader` resolves specifiers in the following priority order:

1. `puer-exec://harness/<id>` → return in-memory generated harness string
2. HTTP(S) URL matching a registered virtual entry pattern → return in-memory code string
3. Any other `http://` or `https://` URL → synchronous blocking `WebClient.DownloadString`
4. All other specifiers → delegate to PuerTS default file-system loader

**Why synchronous HTTP fetch:** PuerTS's `ILoader.ReadFile` is synchronous. The module graph is resolved before `ExecuteModule` returns. Async loading would require a fundamentally different PuerTS integration. For the target use case (localhost dev server), blocking latency is negligible. `WebClient.DownloadString` is acceptable on the main Unity thread for these short requests.

### Decision 3: Entry specifier selection logic

How the harness references the user's entry code depends on input mode and base URL type:

| Mode | import_base_url | Entry specifier in harness | Temp file? |
|------|----------------|---------------------------|------------|
| `--file path.js` | any / none | `source_path` (real abs path) | No |
| `--code` / `--stdin` | filesystem dir | `<base_dir>/__puer_exec_entry_<id>.js` | Yes, written to base_dir |
| `--code` / `--stdin` | HTTP URL | `<http_base>/__puer_exec_entry_<id>` | No (ILoader intercepts) |
| `--code` / `--stdin` | none + no import | N/A (no module resolution needed) | No |
| `--code` / `--stdin` | none + has import | Error: `missing_import_base_url` | — |

"Has import" is detected server-side after the code is received; the error is surfaced as a failed exec result rather than a CLI-level pre-flight check.

### Decision 4: --reset-jsenv-before-exec ordering

`--refresh-before-exec` can trigger C# recompilation, which automatically disposes and recreates the `JsEnv` via `AssemblyReloadEvents`. Resetting the JsEnv before refresh would be wasted work if a compile follows.

Correct order: `refresh (if set) → wait for compile → reset JsEnv (if set) → exec`

`--reset-jsenv-before-exec` is intended specifically for clearing the JS module cache after JS-only file changes (no C# recompile), not as a substitute for refresh.

A standalone `/reset-jsenv` HTTP endpoint is added to the Unity server so the CLI flag can invoke it as a discrete step, and it remains callable independently.

### Decision 5: JsEnv stays singleton; ILoader holds per-request context

`JsEnv` creation is expensive. The ILoader is initialized once with the JsEnv but holds a mutable per-request context slot (current virtual module map + base URL). This slot is set immediately before `ExecuteModule` is called on the main thread and cleared after execution completes.

Since all exec execution is dispatched via `MainThreadActions`, there is no concurrent access to the per-request context. No locking is required.

When `--reset-jsenv-before-exec` fires, a new `JsEnv` is created with a fresh ILoader instance.

## Risks / Trade-offs

- **Synchronous HTTP fetch blocks main Unity thread** → Acceptable for localhost dev-server use; unsuitable for slow remote URLs. Not enforcing a URL allowlist is intentional (product does not own that policy).
- **Temp file for filesystem-base stdin/code mode** → Requires cleanup on exec completion and on JsEnv reset. Cleanup failure leaves orphaned files in the user's import base directory. Mitigation: use a distinctive prefix (`__puer_exec_entry_`) and sweep on next exec to that base.
- **Module cache persists across execs within a JsEnv session** → Imported modules are cached by PuerTS/V8 and will not reload if the file changes. This is the expected behavior; `--reset-jsenv-before-exec` is the documented escape hatch.
- **Error detection for `missing_import_base_url`** → Detecting whether the user's code contains `import` requires a regex scan on the server side, mirroring the existing pattern. A false positive on a string literal containing `import` is tolerable (the user gets an actionable error).
